# Row-level security

For **customer-facing analytics**, each **end-user** should only see the **data they're allowed to see** and nothing more. That's why row-level security is a **first-class citizen** in Embeddable.

There are three ways to do row-level security (RLS) in Embeddable:

1. **Security filters** – the simplest way to ensure row-level security is applied globally across all your data models. Pass filters in the [Tokens API](/deployment/tokens-api) and Embeddable enforces them on every query automatically. If a filter can't be applied, the query fails — so data can never accidentally leak. Read on for details.
2. **SQL-based row-level security** – the recommended approach if you want fine-grained control of row-level security at the model level. Learn more [here](/data-modeling/sql-based-row-level-security).
3. **Access Policies** – a more scalable approach to row-level security if you start using [Views](/data-modeling/views) and want role-based security. Learn more about Access Policies [here](/data-modeling/access-policies).

## Security Filters

Security filters are the simplest way to ensure row-level security is applied globally across all your data models. You pass a `filters` array when requesting a security token, and Embeddable automatically applies those filters to every query in the dashboard session — with no changes to your data models required.

The key advantage is that enforcement is **guaranteed**: if a filter cannot be applied to a query (for example, because there is no join path to the filtered model), the query fails with an error rather than silently returning unfiltered data. This means non-technical team members can build dashboards freely without risking accidental data exposure.

### Adding security filters

Pass a `filters` array in your [Tokens API](/deployment/tokens-api) request:

```ts
fetch('https://api.<region>.embeddable.com/api/v1/security-token', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'Accept': 'application/json',
    'Authorization': `Bearer ${apiKey}`
  },
  body: JSON.stringify({
    // ...
    filters: [{
      member: "customers.id",
      operator: "equals",
      values: [5],
    }]
  })
})
```

Each filter object has three fields:

- **`member`** — the model field to filter on, in `cube.field` notation (e.g. `customers.id`, `orders.region`).
- **`operator`** — how the field is compared to the values. See [Access Policies](/data-modeling/access-policies#3-row_level) for the full list of supported operators.
- **`values`** — an array of values to compare against.

## Testing your security setup

To try out different security configurations in the no-code builder, define presets in your repo at `src/presets/security-contexts.sc.yml` ([here](https://github.com/embeddable-hq/embeddable-boilerplate/blob/main/src/embeddable.com/presets/security-contexts.sc.yml)):

```yaml
- name: Nike
  securityContext:
    orgId: org5
    userId: 23478
  filters:
    - member: customers.id
      operator: equals
      values: [ 23478 ]

- name: Adidas
  securityContext:
    orgId: 23
    userId: cmlidXRlIjoiZ2VvaXBf9sZSJ9LHsiYXR0
  filters:
    - member: customers.id
      operator: equals
      values: [ 'cmlidXRlIjoiZ2VvaXBf9sZSJ9LHsiYXR0' ]
```

Each entry appears under the **"View as"** dropdown in the builder:

Switch between contexts to preview your dashboard as different users.

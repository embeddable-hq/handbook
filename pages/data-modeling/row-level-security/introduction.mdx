# Row-level security

For **customer-facing analytics**, each **end-user** should only see the **data they're allowed to see** and nothing more. That's why row-level security (RLS) is a **first-class citizen** in Embeddable.

There are three ways to do RLS in Embeddable:

1. **[Security filters](/data-modeling/row-level-security/security-filters)** – the simplest way to ensure row-level security is applied globally across all your data models. Pass filters in the [Tokens API](/deployment/tokens-api) and Embeddable enforces them on every query automatically. If a filter can't be applied, the query fails — so data can never accidentally leak.
2. **[SQL-based security](/data-modeling/row-level-security/sql-based-security)** – the recommended approach if you want fine-grained control of (row, schema or table-level) security at the model level.
3. **[Access Policies](/data-modeling/row-level-security/access-policies)** – a more scalable approach to row-level security if you have a lot of models, you are using [Views](/data-modeling/views), or you want role-based security.
